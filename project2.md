### 트러블 슈팅1
제목 : AWS 기반 PWA 애플리케이션 구축 및 개선 과정
#### 문제 상황
1. Express.js 서버와 SMB 서버 연동 시도
내부망에서는 정상적으로 동작했으나, 보안 정책상 외부망에서는 접속이 불가했습니다.
2. PWA 캐시 문제
초기 배포에서는 브라우저 캐시 때문에 최신 코드가 즉시 반영되지 않는 문제가 있었습니다.
Service Worker 버전을 수동으로  업데이트하며 반영했지만, 커밋 누락 시 배포 누락 위험이 발생했습니다.
#### 해결 과정
AWS 기반 서비스 재구성
외부에서도 사용 가능한 서비스를 위해 AWS 생태계를 활용하여 전체 애플리케이션을 재구축했습니다.
Version 2에서 Next.js 프레임워크로 전환하여 유지보수성과 생산성을 높였습니다.
자동 캐시 갱신 기능을 지원하는 PWA 라이브러리를 도입하여 업데이트 안정성을 확보했습니다.
#### 성과
직원들이 외부에서도 안정적으로 시스템을 사용할 수 있어, 업무 효율성이 크게 향상되었습니다.
서버 연동부터 클라우드 환경 구축, CI/CD, 보안 설정까지 엔드투엔드 개발 경험을 쌓을 수 있었습니다.
기능개발 후, 별도로 버전관리를 하지 않고도 자동으로 반영되어 개발경험을 향상시켰습니다. 
#### 배운점
기술 선택의 중요성: 단순히 구현 가능한 기술이 아니라, 서비스의 유지보수성과 확장성을 고려한 기술 선택이 필요함을 체감했습니다.
배포 및 운영 경험: 개발뿐 아니라 배포·운영 과정에서 발생할 수 있는 문제(캐시, 보안, CI/CD 안정성)를 직접 해결하며 성장할 수 있었습니다.
사용자 중심 사고: 단순히 “만들었다”가 아니라, 사용자가 더 편리하게 사용할 수 있도록 개선하는 과정이 서비스 품질에 직결됨을 배웠습니다.

### 트러블 슈팅2

제목 : V3 – Google 생태계로 이전 및 Monorepo 구조 도입
#### 문제 상황
AWS는 안정적이었지만, 월별 과금 부담이 누적되고 있었습니다.
#### 해결 과정
1. Google 생태계로 이전
Firebase Storage로 파일 관리를 전환하였습니다. (기존 S3 대체).
Firebase Functions + Google Sheets 연동으로 인사팀이 직원별 사용 금액을 시트에서 직접 확인할 수 있도록 구현하였습니다.
App Script를 통해 Functions를 호출, 데이터가 자동으로 업데이트되도록 설정하였습니다.
2. Monorepo + pnpm 도입 검토
배경: 추후 어드민 페이지 개발 가능성과 이때 사용될 공용 패키지를 고려한 결과, 적절한 구조라고 판단하였습니다.
높은 유지보수성과 확장성을 고려해 패키지 단위로 관리하고자  pnpm을 채택하였습니다.  
공통 패키지에는 shadcn/ui 기반의 UI 컴포넌트와 공용 유틸 함수를 모아 pnpm의 장점을 활용하였습니다.
#### 성과
비용 절감: AWS 대비 Firebase로 이전하여 운영비를 0원으로 줄였습니다. 매월 12,000원(기존) → 현재:0원
관리 효율화: 인사팀이 별도 툴 없이 Google Sheets에서 직접 데이터를 확인 할 수 있습니다.
#### 배운점
단순한 기능 구현을 넘어서 비용 최적화, 유지보수성, 확장성까지 고려한 아키텍처 설계의 중요성을 체감할 수 있었습니다.
pnpm + monorepo 구조 검토를 통해 멀티 애플리케이션 관리 전략에 대한 이해를 확장시켰던 경험이였습니다.
